주요 기능 구현

FEAT · 01
BatteryMeasurement
상태 기반 전류 유효성 검증

충전 중 전류는 음수(−12A~0A), 주행 중은 양수(0~15A)로 기준이 반대입니다. bmsState를 참조해 분기함으로써 단순 절댓값 비교로는 구현할 수 없는 문맥 의존 유효성 검증을 실현했습니다. 7채널 circular buffer 기반 5-sample 이동 평균 필터로 센서 노이즈를 억제합니다.

SWC_BatteryMeasurement.c
/* 상태별 전류 방향 분기 검증 */
if (bmsState == BMS_STATE_CHARGE) {
    if (packCurrent < CURRENT_CHARGE_MIN
     || packCurrent > 0)
        measurementValid = FALSE;
} else {
    if (packCurrent < 0
     || packCurrent > CURRENT_DISCHARGE_MAX)
        measurementValid = FALSE;
}
/* circular buffer 이동 평균 */
voltageBuffer1[filterIndex] = cellVoltage1;
filterIndex = (filterIndex + 1u) % FILTER_SIZE;
FEAT · 02
FaultDiagnosis
비트마스크 다중 Fault 추적

5개 Fault 유형을 각각 독립 비트(0x0001~0x0010)로 OR 합산합니다. 과전압 + 과온도 동시 발생 시 faultCode = 0x0009로 정확히 식별할 수 있어, boolean 하나로는 불가능한 다중 동시 Fault 진단이 가능합니다. 새 Fault 추가 시 비트 정의만 확장하면 되어 확장성도 확보했습니다.

SWC_FaultDiagnosis.c
#define FAULT_OVERVOLTAGE  (0x0001u)
#define FAULT_UNDERVOLTAGE (0x0002u)
#define FAULT_OVERCURRENT  (0x0004u)
#define FAULT_OVERTEMP     (0x0008u)
#define FAULT_UNDERTEMP    (0x0010u)

/* OR 합산으로 다중 Fault 누적 */
if (overVoltageFlag == TRUE)
    faultCode |= FAULT_OVERVOLTAGE;
if (overTempFlag == TRUE)
    faultCode |= FAULT_OVERTEMP;
/* faultCode == 0x0009 → 과전압+과온도 동시 */
FEAT · 03
BMSStateMachine
5-State FSM + Fault 최우선 전이

INIT → STANDBY → DRIVE / CHARGE 전이 구조에서 어느 상태에서도 faultStatus 감지 시 즉시 FAULT로 전이합니다. static 변수로 상태를 유지하고, 상태에 따라 RelayEnable·BalancingEnable 신호를 자동 생성합니다. FAULT 상태에서는 Fault 해소 + 키 입력이 모두 확인돼야만 복귀합니다.

SWC_BMSStateMachine.c
case BMS_STATE_STANDBY:
  if (faultStatus == TRUE)
    currentState = BMS_STATE_FAULT; // 최우선
  else if (chargingActive == TRUE)
    currentState = BMS_STATE_CHARGE;
  else if (keyOn == TRUE)
    currentState = BMS_STATE_DRIVE;
  break;

/* DRIVE·CHARGE에서만 릴레이 허용 */
if (relayRequest && !faultStatus
 && (state==DRIVE||state==CHARGE))
  relayEnable = TRUE;
FEAT · 04
RelayDecision
4중 안전 조건 AND 검증

BMSStateMachine이 이미 RelayEnable을 제어하지만, RelayDecision에서 Fault·상태·요청을 한 번 더 독립 검증합니다. 단일 컴포넌트 실패가 릴레이 오작동으로 이어지지 않도록 의도적으로 중복 검증 구조(Defense in Depth)를 설계했습니다.

SWC_RelayDecision.c
/* 4중 AND — 하나라도 실패 시 차단 */
if (faultStatus  == FALSE
 && relayRequest  == TRUE
 && relayEnable   == TRUE
 && (bmsState == BMS_STATE_DRIVE
  || bmsState == BMS_STATE_CHARGE))
{
    relayCommand = TRUE;
} else {
    relayCommand = FALSE; // fail-safe
}
FEAT · 05
VCU CAN Tx Handler
VCU → BMS 명령 송신

VCUTxHandler는 KeyOn·ChargingActive·RelayRequest 3개 명령을 CAN 프레임으로 BMS에 전송합니다. RTE 내부에서 boolean으로 관리하던 값을 CAN 시그널 타입(uint8/uint16)으로 명시적 변환해 송신합니다.

SWC_VCUTxHandler.c
/* boolean → uint8 변환 후 CAN 송신 */
uint8 keyOn_u8    = (keyOn    == TRUE) ? 1U : 0U;
uint8 charging_u8 = (charging == TRUE) ? 1U : 0U;
uint16 relayRequest_u16 =
    (relayRequest_bool == TRUE) ? 1u : 0u;

Rte_Write_ECU2_Msg_OE1_Sig1(keyOn_u8);
Rte_Write_ECU2_Msg_OE1_Sig2(charging_u8);
Rte_Write_ECU2_Msg_OE1_Sig3(relayRequest_u16);
FEAT · 06
VCU CAN Rx Handler
BMS → VCU 데이터 수신

VCUCANRxHandler는 BMS로부터 수신한 SOC·RelayCommand·FaultStatus·BMSState 5개 신호를 읽어 VCU 내부 RTE 포트에 분배합니다. CAN 시그널 타입(uint8)을 내부 RTE 타입(boolean)으로 복원하는 변환을 명시적으로 처리했습니다.

SWC_VCUCANRxHandler.c
/* CAN 수신 후 uint8 → boolean 복원 */
Rte_Read_ECU1_Msg_OE2_Sig2(&relayCommand_raw);
Rte_Read_ECU1_Msg_OE2_Sig4(&fault_raw);

Rte_Write_P_RelayRequestData_RelayCommand(
    (relayCommand_raw == 1U) ? TRUE : FALSE);
Rte_Write_P_RelayRequestData_FaultStatus(
    (fault_raw == 1U) ? TRUE : FALSE);
Rte_Write_P_SOC_SOC((uint16)soc);
Rte_Write_P_RelayRequestData_BMSState(bmsState);
FEAT · 05
VCU CAN Tx / Rx Handler
VCU 명령 송신 & BMS 데이터 수신

VCUTxHandler는 KeyOn·ChargingActive·RelayRequest 3개 명령을 boolean → uint8 변환 후 CAN으로 BMS에 전송합니다. VCUCANRxHandler는 BMS로부터 수신한 SOC·RelayCommand·FaultStatus·BMSState를 uint8 → boolean으로 복원해 VCU 내부 RTE 포트에 분배합니다. 양방향 모두 CAN 시그널 타입과 내부 RTE 타입 간 변환을 명시적으로 처리했습니다.

SWC_VCUTxHandler.c / VCUCANRxHandler.c
/* VCU Tx: boolean → uint8 변환 후 송신 */
uint8 keyOn_u8 = (keyOn == TRUE) ? 1U : 0U;
Rte_Write_ECU2_Msg_OE1_Sig1(keyOn_u8);
Rte_Write_ECU2_Msg_OE1_Sig3(relayRequest_u16);

/* VCU Rx: uint8 → boolean 복원 후 분배 */
Rte_Read_ECU1_Msg_OE2_Sig2(&relayCommand_raw);
Rte_Write_P_RelayRequestData_RelayCommand(
    (relayCommand_raw == 1U) ? TRUE : FALSE);
Rte_Write_P_RelayRequestData_FaultStatus(
    (fault_raw == 1U) ? TRUE : FALSE);
03

설계 의사결정

RTE Port
Interface

SWC 간 완전한 인터페이스 분리

각 SWC가 RTE 포트를 통해서만 데이터를 교환하도록 설계해 모듈 간 의존성을 제거했습니다. 개별 SWC의 독립적인 단위 테스트가 가능하고, 특정 SWC 교체 시 타 SWC 수정이 불필요한 구조입니다. 팀 개발에서 인터페이스만 합의하면 구현을 병렬로 진행할 수 있었습니다.

Bitmask
Fault Code

boolean이 아닌 uint16으로 다중 Fault 동시 추적

boolean 하나로는 어떤 Fault가 동시에 발생했는지 구분할 수 없습니다. uint16 faultCode로 5개 플래그를 독립 비트에 매핑해 다중 동시 Fault를 정밀 추적하고, 새 Fault 추가 시 기존 로직 수정 없이 비트 정의만 확장하면 되는 구조를 확보했습니다.

Defense
in Depth

릴레이 이중 검증 — 단일 컴포넌트 실패를 방어

BMSStateMachine에서 RelayEnable을 제어하지만, RelayDecision에서 Fault·상태·요청을 한 번 더 독립 검증합니다. 어느 한 SWC에 버그가 있어도 릴레이 오작동이 발생하지 않도록 의도적으로 중복 검증 구조를 설계했습니다.

Context-Aware
Validation

BMS 상태를 참조한 전류 유효성 분기

충전·주행 모드에서 전류 방향이 반대이므로 단순 절댓값 비교로는 올바른 유효성 검증이 불가능합니다. bmsState를 참조해 분기하는 구조로, 시스템 상태 맥락을 반영한 측정 유효성 판단을 구현했습니다.

실물 보드 검증

TRK-MPC5606B 보드에서 BMS SWC를 동작시키며 충전 중 과전압 감지 → Fault 전이 → 릴레이 차단 시나리오를 직접 검증했습니다. LED01~03은 셀 밸런싱 필요 여부, LED04는 릴레이 상태와 SOC를 나타냅니다.
INIT 상태 — 릴레이 Open, LED 전체 소등
Charge 상태 — 릴레이 Close, LED04 점등
Cell2 전압 상승 — LED02 점등 (밸런싱 필요)
Fault 전이 — 릴레이 Open, LED04 소등
SOC 낮음 — PWM Duty 낮아 LED 어둡게 점등
SOC 높음 — PWM Duty 높아 LED 밝게 점등

01

시스템 아키텍처

BMS의 SWC 분리 원칙을 참고해 Driver / Logic / Control 3계층으로 모듈을 분리했습니다. 각 레이어가 인터페이스 포인터로만 통신하도록 설계해, 센서 교체 시 Driver 레이어만, 알고리즘 변경 시 Logic 레이어만 수정하면 됩니다.
Layer · 01 — Driver
UltrasonicDriver
HC-SR04 Trig/Echo 펄스 제어, pulseIn으로 왕복 시간 측정, 거리 계산(duration/2 × 0.034). 측정 실패(타임아웃) 시 −1.0f 반환으로 상위 레이어에 에러 전달
Layer · 01 — Driver
LineTrackingDriver
3채널(좌/중/우) 디지털 입력 읽기, HIGH = 검은 라인 위. LineState 구조체로 반환해 상위 레이어가 핀 번호 없이 라인 위치만 참조하도록 추상화
Layer · 01 — Driver
IRDriver
IRremote 라이브러리 래핑, 수신 코드 디코딩 및 resume() 호출. 상위 레이어에 available() / read() 인터페이스만 노출해 라이브러리 종속성을 캡슐화
Layer · 02 — Logic
ObstacleDetection / LineTracking
센서 데이터를 해석해 주행 판단. ObstacleDetection은 감지(detect) + 회피(avoid) 분리. LineTracking은 3채널 상태 조합으로 방향 결정

주요 기능 구현

FEAT · 01
IR 리모컨 모드 전환
MANUAL / LINE_TRACKING

IR 수신 코드를 디코딩해 버튼 '2'는 LINE_TRACKING 모드, 버튼 '5'는 MANUAL 모드로 전환합니다. ModeManager가 현재 모드를 관리하고, 메인 루프에서 모드에 따라 자율주행 또는 정지를 분기합니다. IRDriver로 라이브러리 종속성을 캡슐화해 상위 레이어는 available() / read() 인터페이스만 사용합니다.

main.cpp
if (irDriver.available()) {
    unsigned long code = irDriver.read();
    /* 버튼 5 → MANUAL, 버튼 2 → LINE_TRACKING */
    if      (code == 3810328320)
        modeManager.setMode(Mode::MANUAL);
    else if (code == 3877175040)
        modeManager.setMode(Mode::LINE_TRACKING);
}
/* 모드 기반 분기 실행 */
if (modeManager.getMode() == Mode::LINE_TRACKING)
    obstacleLogic.detect() ? avoid() : lineLogic.update();
FEAT · 02
ObstacleDetection
우선순위 기반 장애물 회피

20cm 이내 장애물 감지 시 라인 트래킹보다 우선 처리합니다. 회피 시퀀스는 정지(500ms) → 후진(500ms) → 우회전(600ms) → 전진(500ms) 순서로 진행합니다. 측정 타임아웃(30ms) 시 −1.0f를 반환해 센서 이상을 상위 레이어에 명시적으로 전달합니다.

ObstacleDetection.cpp
bool ObstacleDetection::detect() {
    float distance = _ultrasonicDriver->getDistance();
    return (distance >= 0.0f
         && distance <= OBSTACLE_DISTANCE_CM);
}

void ObstacleDetection::avoid() {
    _motorControl->stop();   delay(500);
    _motorControl->backward(); delay(OBSTACLE_BACK_MS);
    _motorControl->turnRight(); delay(OBSTACLE_TURN_MS);
    _motorControl->forward();  delay(500);
}
FEAT · 03
LineTracking
3채널 센서 조합 방향 판단

좌/중/우 3채널 디지털 입력 조합으로 현재 라인 위치를 판단합니다. 중앙 센서 활성 시 전진, 좌측만 활성 시 좌회전, 우측만 활성 시 우회전, 모두 비활성 시 정지합니다. LineTrackingDriver가 핀 제어를 추상화해 Logic 레이어는 LineState 구조체만 참조합니다.

LineTracking.cpp
void LineTracking::update() {
    LineState state = _lineDriver->read();

    if      (state.middle)
        _motorControl->forward();
    else if (state.left && !state.right)
        _motorControl->turnLeft();
    else if (!state.left && state.right)
        _motorControl->turnRight();
    else
        _motorControl->stop(); // 라인 이탈
}
FEAT · 04
MotorControl
L298N 추상화 인터페이스

L298N의 IN1~IN4 핀 제어와 ENA/ENB PWM 속도 제어를 forward() / backward() / turnLeft() / turnRight() / stop() 인터페이스로 추상화했습니다. 상위 레이어는 핀 번호나 H-Bridge 동작 방식을 알 필요 없이 방향 함수만 호출합니다.

MotorControl.cpp
void MotorControl::turnLeft() {
    /* 오른쪽 전진, 왼쪽 역회전 */
    digitalWrite(PIN_MOTOR_IN1, LOW);
    digitalWrite(PIN_MOTOR_IN2, HIGH);
    digitalWrite(PIN_MOTOR_IN3, LOW);
    digitalWrite(PIN_MOTOR_IN4, HIGH);
    analogWrite(PIN_MOTOR_ENA, MOTOR_SPEED_NORMAL);
    analogWrite(PIN_MOTOR_ENB, MOTOR_SPEED_NORMAL);
}
03

설계 의사결정

3-Layer
Architecture

Driver / Logic / Control 계층 분리

BMS의 SWC 분리 원칙을 Arduino 환경에 적용했습니다. 센서 교체 시 Driver 레이어만, 제어 알고리즘 변경 시 Logic 레이어만 수정하면 됩니다. 각 모듈이 인터페이스 포인터로만 통신해 구현 세부사항을 캡슐화했습니다.

Priority
Control

장애물 감지 최우선 처리

라인 트래킹과 장애물 감지가 동시에 입력될 때 충돌 방지를 최우선으로 처리합니다. obstacleLogic.detect() 결과를 먼저 확인해 true이면 avoid(), false일 때만 lineLogic.update()를 실행하는 명확한 우선순위 흐름을 구현했습니다.

Error
Propagation

센서 이상값 명시적 전달 (−1.0f)

초음파 센서 타임아웃(30ms 초과) 시 −1.0f를 반환해 상위 레이어에 측정 실패를 명시적으로 전달합니다. 0을 반환하면 "0cm 장애물"로 오인될 수 있어, 유효 범위 외 값으로 에러 상태를 명확히 구분했습니다.

Mode
Manager

IR 리모컨 기반 모드 전환 분리

모드 관리를 ModeManager 클래스로 분리해 메인 루프의 관심사를 단순화했습니다. 입력(IR 수신), 상태 관리(ModeManager), 동작 실행(Logic 레이어)을 각각 독립 책임으로 분리하는 구조입니다.

06

배운 점

🏗️
설계 원칙의 이식성
BMS에서 익힌 SWC 분리·인터페이스 우선 원칙이 Arduino 환경에도 그대로 적용됐습니다. 플랫폼이 달라져도 설계 사고방식은 동일하게 작동함을 확인했습니다.
⚡
H/W–S/W 인터페이스
L298N 핀 제어, pulseIn 타이밍, PWM duty cycle과 실제 모터 동작의 관계를 직접 다루며 하드웨어–소프트웨어 경계를 이해했습니다.
🎛️
에러 상태 명시적 처리
센서 타임아웃을 0이 아닌 −1.0f로 반환하는 설계가 왜 중요한지, 모호한 에러 표현이 상위 레이어에서 어떤 오작동을 유발할 수 있는지 직접 겪으며 이해했습니다.
🔄
우선순위 기반 제어
안전 관련 조건(장애물 회피)을 항상 먼저 처리하는 제어 흐름 설계가 BMS의 Fault 최우선 전이 원칙과 동일한 맥락임을 확인했습니다.
📦
캡슐화의 실용성
IRremote 라이브러리 종속성을 IRDriver로 래핑하니 상위 레이어 코드가 단순해지고 라이브러리 교체 시 영향 범위가 Driver 레이어로 한정됐습니다.
🚗
실제 하드웨어 디버깅
코드가 맞아도 배선, 전원, 센서 노이즈로 동작하지 않는 경우를 직접 경험하며, 하드웨어와 소프트웨어를 분리해 디버깅하는 방법을 익혔습니다.
